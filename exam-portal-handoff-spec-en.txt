PROJECT HANDOFF SPEC
ExamPortal

1. PROJECT IDENTITY

Project name: ExamPortal
Repository/folder name: exam-portal
Frontend: React
Backend: Spring Boot
Database: SQL Server

Current implementation priority:
Focus on building a strong and complete online examination system first.
Challenge mode and leaderboard are future extensions and should NOT be included in the main implementation scope right now.


2. PROJECT OVERVIEW

ExamPortal is a full-stack web application for online multiple-choice examinations.

This is not just a management website. The core business happens directly inside the system:
- instructors create questions and exams
- students log in and take exams directly on the website
- the system automatically grades submissions
- students view their results
- instructors and admins review attempts and statistics

Main business focus:
- exam creation
- exam scheduling
- online exam participation
- automatic grading
- result management

Future extension:
- public challenge mode
- leaderboard and ranking


3. CORE SCOPE FOR CURRENT VERSION

The current version must focus on exam features only.

Included:
- authentication and authorization
- user management
- subject/topic management
- question bank management
- exam management
- exam session/scheduling management
- online exam participation
- automatic grading
- result review
- basic statistics

Excluded from current version:
- challenge mode
- leaderboard
- essay questions
- coding questions
- manual grading
- webcam proctoring
- advanced anti-cheat
- realtime features via WebSocket


4. MAIN ROLES

4.1. Student
- register account
- log in / log out
- view available exams
- start an exam
- answer multiple-choice questions online
- submit exam
- receive automatic score
- view detailed result
- view own attempt history
- update own profile

4.2. Instructor
- manage question bank
- manage subjects/topics

- create and manage exam sessions

- review results
- view basic statistics

4.3. Admin
- manage all users
- assign roles
- lock/unlock accounts
- manage subjects/topics
- view all exams and sessions
- view system-wide dashboard/statistics


5. QUESTION TYPE RULES

The first version should support only:
- multiple-choice questions
- single correct answer
- exactly 4 options per question: A, B, C, D

Do not include:
- essay questions
- multiple correct answers
- coding questions

7. MAIN BUSINESS FLOWS

7.1. Instructor creates an exam
1. Instructor logs in
2. Instructor creates subjects/topics if needed
3. Instructor creates questions in question bank
4. Instructor creates an exam
5. Instructor adds questions to the exam
6. Instructor configures duration, score, and visibility
7. Instructor publishes the exam

7.2. Instructor creates an exam session
1. Instructor selects an existing exam
2. Instructor creates an exam session
3. Instructor sets:
   - open time
   - close time
   - duration
   - attempt limit
4. Instructor activates the session

7.3. Student takes an exam
1. Student registers or logs in
2. Student views available exam sessions
3. Student opens an exam session detail page
4. Student clicks Start Exam
5. System creates an attempt
6. Student answers multiple-choice questions
7. Student submits the exam or system auto-submits when time ends
8. System grades the attempt automatically
9. Student views result

7.4. Admin monitors platform
1. Admin logs in
2. Admin opens dashboard
3. Admin reviews users, exams, sessions, attempts, and statistics
4. Admin manages system data if needed

17. SCORING LOGIC

Recommended first-version scoring:
- each correct answer = 1 point
- wrong answer = 0 point
- unanswered = 0 point

Display options:
- raw score = correct answer count
- normalized score = correct_answers / total_questions * 10


18. VALIDATION RULES

18.1. User validation
- email must be unique
- password must meet minimum length
- role must be valid

18.2. Question validation
- question must have content
- exactly 4 options
- exactly 1 correct option

18.3. Exam validation
- exam must contain at least 1 question before publish
- duration must be positive

18.4. Exam session validation
- close_time must be after open_time
- duration must be positive
- max_attempts must be valid

18.5. Attempt validation
- student can only start if session is active
- student cannot exceed max_attempts
- attempt cannot be submitted twice
